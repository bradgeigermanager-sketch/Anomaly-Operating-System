# **1. VERBOSE HUMAN DOCUMENTATION**  
## **AOS Iterative Workspace — User Guide (Human Edition)**

### **Purpose of the Workspace**
The AOS workspace is a structured notebook for analyzing anomalies using the four‑stage loop:

1. **Sense** — What was observed?  
2. **Triage** — How serious is it? What surfaces are affected?  
3. **Explain** — What hypotheses explain the anomaly?  
4. **Adapt** — What should change to prevent recurrence?

Each cycle is stored as an **iteration**, forming a timeline of reasoning.

### **How the Workspace Works**
- You can type directly into the four sections.  
- You can paste JSON into the injection box to auto‑populate fields.  
- You can send data via URL parameters or URL‑encoded JSON.  
- You can stream data via WebSocket.  
- You can send an iteration to an LLM endpoint and receive a new iteration back.  
- You can run **LLM auto‑iteration loops**, where the LLM generates multiple sequential iterations.

### **What an Iteration Represents**
Each iteration is a snapshot of your reasoning at a point in time:

- What you sensed  
- How you triaged it  
- What explanations you considered  
- What adaptations you proposed  

Iterations accumulate into a **timeline**, allowing you to:

- Navigate backward and forward  
- Compare reasoning steps  
- Export the entire chain  
- Use it as a training or audit artifact  

### **When to Use the Workspace**
Use it whenever you need to:

- Analyze anomalies  
- Perform structured reasoning  
- Collaborate with an LLM  
- Maintain a traceable chain of thought  
- Build a dataset of anomaly‑analysis examples  
- Conduct investigations or postmortems

# **3. LLM‑SPECIFIC USAGE GUIDE**

This section tells an LLM **exactly how to behave** when interacting with the workspace.

## **LLM Responsibilities**
When receiving an iteration, the LLM should:

1. **Read the four fields**: sense, triage, explain, adapt  
2. **Produce a new iteration** that:  
   - Advances the reasoning  
   - Adds new insights  
   - Refines hypotheses  
   - Suggests next steps  
3. **Return JSON ONLY**, with the fields:  
   - `domain`  
   - `sense`  
   - `triage`  
   - `explain`  
   - `adapt`  

## **LLM Output Format**
The LLM must return:

```json
{
  "domain": "system_logs",
  "sense": "New observations...",
  "triage": "Updated severity...",
  "explain": "Refined hypotheses...",
  "adapt": "Next steps..."
}
```

No extra text. No markdown. No commentary.

## **LLM Should NOT**
- Repeat the previous iteration  
- Produce empty fields  
- Produce unrelated content  
- Break JSON format  
- Add fields not recognized by the workspace

# **4. PSEUDOCODE SPECIFICATIONS**

## **Iteration Creation**
```pseudo
function buildRecord():
    record = {
        iteration: iterations.length + 1
        timestamp: now()
        domain: UI.domain
        sense: UI.sense
        triage: UI.triage
        explain: UI.explain
        adapt: UI.adapt
    }
    return record
```

## **Auto‑Advance on URL Ingestion**
```pseudo
if URL has params:
    populate fields
    record = buildRecord()
    iterations.append(record)
    clear fields
```

## **WebSocket Ingestion**
```pseudo
on websocket message:
    data = parse JSON(message)
    populate fields with data
    record = buildRecord()
    iterations.append(record)
    clear fields
```

## **LLM Auto‑Iteration Loop**
```pseudo
payload = buildRecord()

for i in 1..N:
    response = POST(llmUrl, payload)
    data = parse JSON(response)
    populate fields with data
    record = buildRecord()
    iterations.append(record)
    payload = record
```
